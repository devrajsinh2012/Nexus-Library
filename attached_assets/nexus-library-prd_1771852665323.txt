2. Problem Statement & Opportunity
2.1 Current State of Library Software
Traditional Library Management Systems (LMS) such as Koha, Evergreen, and proprietary systems like Symphony suffer from a common set of critical deficiencies that Nexus is designed to eliminate:


* Outdated, early-2000s UI paradigms that confuse patrons and require staff training
* No AI assistance for discovery, recommendations, or cataloging
* High licensing fees ($10,000–$100,000+/year) or complex self-hosting costs
* Poor mobile experience — not responsive or PWA-capable
* Siloed data with no open APIs or webhook systems
* No real-time features — holds, availability, and notifications are batch-processed
* Zero integration with modern content ecosystems (eBooks, audiobooks, streaming)
* Inaccessible to small and community libraries due to cost and complexity


2.2 The Nexus Opportunity
By combining Supabase (free tier: 500 MB DB, 1 GB storage, 2 GB bandwidth/month, unlimited API calls), NVIDIA NIM free credits, Vercel free hosting, and open library APIs, Nexus can deliver enterprise-grade functionality at zero cost — making it accessible to any library worldwide.


Nexus targets three user segments simultaneously: (1) Public & community libraries needing a free, modern replacement for legacy software. (2) University libraries wanting AI-powered research discovery. (3) Corporate / private libraries needing analytics and integration capabilities.

4.2 System Architecture
High-Level Architecture Description
The system follows a modern Jamstack + edge architecture with three primary tiers:


* Presentation Tier: Next.js 14 App Router deployed on Vercel Edge Network. Server Components handle initial render, Client Components manage interactivity. ISR caches catalog pages with 60-second revalidation.
* API Tier: Next.js API Routes + Supabase Edge Functions handle business logic. Supabase Row Level Security (RLS) enforces data access at the database level, eliminating the need for a separate middleware auth layer.
* Data Tier: Supabase PostgreSQL with pgvector extension enabled for semantic search embeddings. Supabase Realtime provides WebSocket connections for live updates (availability, notifications, chat).


AI Pipeline Architecture
All AI features route through a single unified AI service layer (lib/ai/nexus-ai.ts) which abstracts the NVIDIA NIM Kimi K2 API. The pipeline:
12. Request arrives at Next.js API Route /api/ai/[feature]
13. Rate limiter checks Supabase table (ai_request_log) — max 100 requests/patron/day
14. Context builder assembles relevant data (user history, book metadata, catalog state)
15. Prompt template injects context into structured Kimi K2 prompt
16. NVIDIA NIM API called via streaming fetch with 30-second timeout
17. Response streamed to client via Server-Sent Events for real-time typewriter effect
18. Response cached in Supabase (ai_response_cache) with 24-hour TTL to minimize API calls

6. AI Features (NVIDIA NIM Kimi K2 Integration)
All AI features use the NVIDIA NIM API endpoint: https://integrate.api.nvidia.com/v1/chat/completions with model: "moonshotai/kimi-k2-instruct". API key stored in Supabase Vault as NIM_API_KEY. Streaming is enabled for all user-facing features. Implement exponential backoff retry (3 attempts) for all API calls.
	

6.1 AI Feature Inventory
AI Feature
	Description & Implementation Notes
	NexusBot — AI Librarian Chat
	Full conversational AI assistant embedded in sidebar and mobile drawer. Maintains session context in ai_chats table. System prompt includes library catalog summary, patron reading history, and current policies. Supports: book queries, policy questions, reading recommendations, research guidance, homework help. Uses streaming SSE for real-time response rendering with typewriter animation.
	Semantic Search
	Natural-language book search beyond keyword matching. User query is embedded using a lightweight sentence-transformer (via Hugging Face Inference API free tier) then compared against book embeddings stored in pgvector. Returns top-N results ranked by cosine similarity + keyword relevance hybrid score.
	AI Book Recommendations
	Personalized "For You" shelf. Kimi K2 analyzes patron's checkout history, ratings, browsing patterns, and explicit preferences to generate a ranked recommendation list. Recomputed daily via Supabase Cron job and cached in patron's ai_reading_profile column.
	Auto-Cataloging AI
	When a librarian scans an ISBN, Kimi K2 automatically: writes a professional catalog description, generates subject headings, assigns reading level/audience, suggests Dewey Decimal class, extracts key themes and tags. Reduces cataloging time from 15 minutes to under 60 seconds.
	AI Book Summarizer
	One-click AI-generated book summary on any book detail page. Kimi K2 generates a 3-paragraph structured summary: (1) What the book is about, (2) Why it matters / key themes, (3) Who should read it. Summary cached in ai_summary column for 90 days.
	AI Reading Streak Coach
	Personalized motivational messages and reading goals. Kimi K2 analyzes reading patterns and generates daily personalized encouragement, weekly reading reports, and adaptive goal suggestions. Delivered as push notification + in-app card.
	Smart Search Suggestions
	As patron types in search bar, AI generates intelligent autocomplete suggestions that go beyond simple prefix matching. Suggestions include: author variations, related subjects, similar title patterns, genre-specific query refinements.
	AI Acquisition Advisor
	Analyzes hold queues, patron search queries with zero results, community request trends, and genre gaps in catalog. Generates weekly acquisition recommendation report for librarians with priority scores and estimated demand.
	Review Sentiment Analysis
	When a patron submits a review, Kimi K2 analyzes sentiment, extracts key themes mentioned, flags potentially inappropriate content, and generates a one-line "patron verdict" summary displayed under the review.
	AI-Powered Fine Waiver Assistant
	Analyzes patron history (length of membership, total books read, previous fines, reason for late return) and recommends whether a fine waiver is appropriate, with reasoning. Final decision remains with librarian.
	AI Reading Level Detector
	For books lacking Lexile/AR data, Kimi K2 analyzes title, description, author, genre, and publication year to estimate reading level and appropriate age range.
	NLP Event Description Generator
	Librarians enter basic event details; Kimi K2 generates a full, engaging event description formatted for the web, email newsletter, and social media (three format variants).
	AI Demand Forecasting
	Predicts which books will have high demand in the next 30 days based on: seasonality, school curriculum schedules, trending topics, current hold-to-copy ratios, and patron search trends.
	Research Paper Assistant
	For university/researcher patrons. Given a research topic, Kimi K2 generates: a structured literature review outline, key search terms to try in the catalog, related subject headings, and citation formatting guidance.
	

6.2 NVIDIA NIM API Integration Code Specification
The AI agent must implement the following module at lib/ai/nexus-ai.ts:


File: lib/ai/nexus-ai.ts — Exports: streamAIResponse(), generateAISummary(), buildCatalogContext(), buildPatronContext(), rateLimitCheck(). All functions are async and throw typed NexusAIError on failure. Never expose API key to client-side code.
	

* Base URL: https://integrate.api.nvidia.com/v1
* Model: moonshotai/kimi-k2-instruct
* Default max_tokens: 2048 for summaries, 4096 for chat, 512 for tags
* Temperature: 0.3 for factual tasks (cataloging, summaries), 0.7 for conversational chat, 0.9 for creative (event descriptions)
* Always include a structured system prompt that defines the AI as "Nexus AI, the intelligent librarian assistant"
* Implement streaming via ReadableStream and Server-Sent Events in Next.js Route Handlers
* Store all prompts/responses in ai_chats table for debugging and improvement
* Implement daily token usage tracking in Supabase to stay within free tier limits

8. UI/UX Design System & Requirements
8.1 Design Philosophy
Nexus UI is designed to feel like a premium consumer product — think Notion meets Linear meets a luxury bookstore. Every screen must pass the "first impression test": a non-technical patron should feel delighted and immediately understand what to do. The design system is built on three pillars:


* Clarity: Information hierarchy is always obvious. The most important action on any screen is the most visually prominent. No cognitive overload.
* Delight: Micro-interactions, smooth transitions, and subtle animations reward engagement without distracting from function. Every hover, click, and state change has a thoughtful animation.
* Inclusivity: Beautiful design that is also fully accessible. No contrast failures. No keyboard traps. No motion sickness from excessive animation (respects prefers-reduced-motion).

Use: aesthetic fallback cover images when no ISBN cover is available. Cache photo_id in book.unsplash_cover_id. Always credit photographer as required by Unsplash guidelines.

	API key via env var RESEND_API_KEY. Free: 100 emails/day, 3,000/month. Use: hold-ready notifications, overdue reminders, welcome email, event confirmations, weekly reading reports. Templates stored in Supabase, rendered with React Email components. Queue emails via Supabase Database Webhooks + Supabase Edge Functions.
	Wikipedia API

	MeiliSearch Cloud (free)
	API keys via env vars MEILISEARCH_HOST + MEILISEARCH_ADMIN_KEY. Free: 100,000 docs. Use: fast full-text catalog search as complement to Supabase full-text. Index: books (title, authors, description, ai_tags). Sync on book create/update via Supabase webhook.
	Standard Ebooks (standardebooks.org)

	Leaflet.js + OpenStreetMap

10. Security, Auth & Supabase RLS Policies
10.1 Authentication Flow
Authentication is handled entirely by Supabase Auth. The system supports:


* Email + Password (primary method for patrons and staff)
* Magic Link (passwordless email login for patrons)
* OAuth: Google and GitHub (optional, for patron convenience)
* Staff accounts created by admins via invite — no self-registration for librarian+ roles
* JWT tokens stored in httpOnly cookies (not localStorage) via @supabase/ssr
* Session refresh: automatic via Supabase middleware in Next.js middleware.ts
* MFA (TOTP): Required for admin and superadmin roles via Supabase Auth MFA


10.2 Row Level Security Policies
Every table in Supabase must have RLS enabled. The following policies are MANDATORY — the AI agent must implement all of them before any table is considered done.
	

Table / Policy
	Policy Definition
	books — Public Read
	POLICY: SELECT for authenticated and anon. USING: is_active = true. All authenticated users and anonymous visitors can read active books.
	books — Librarian Write
	POLICY: INSERT, UPDATE, DELETE for authenticated. USING/CHECK: auth.jwt()->>role IN (librarian, admin, superadmin). Only library staff can modify catalog.
	users — Own Profile Read
	POLICY: SELECT for authenticated. USING: id = auth.uid(). Patrons can only read their own profile row.
	users — Staff Read All
	POLICY: SELECT for authenticated. USING: auth.jwt()->>role IN (librarian, admin, superadmin). Staff can read all patron profiles.
	users — Own Profile Update
	POLICY: UPDATE for authenticated. USING: id = auth.uid(). CHECK: role = OLD.role (patrons cannot self-escalate role).
	loans — Own Loans Read
	POLICY: SELECT for authenticated. USING: patron_id = auth.uid() OR auth.jwt()->>role IN (librarian, admin, superadmin).
	loans — Librarian Write
	POLICY: INSERT, UPDATE for authenticated. USING: auth.jwt()->>role IN (librarian, admin, superadmin).
	holds — Own Holds Read/Write
	POLICY: SELECT, INSERT for authenticated. USING: patron_id = auth.uid(). Patrons can create and view their own holds.
	holds — Cancel Own Hold
	POLICY: UPDATE for authenticated. USING: patron_id = auth.uid(). CHECK: status = cancelled (patrons can only cancel, not change to other statuses).
	reviews — Public Read Approved
	POLICY: SELECT for anon and authenticated. USING: is_approved = true. Only approved reviews are publicly visible.
	reviews — Own Review Write
	POLICY: INSERT, UPDATE, DELETE for authenticated. USING: patron_id = auth.uid().
	ai_chats — Own Chats Only
	POLICY: ALL for authenticated. USING: patron_id = auth.uid(). AI conversation history is strictly private.
	staff_activity_log — Admin Read
	POLICY: SELECT for authenticated. USING: auth.jwt()->>role IN (admin, superadmin). Only admins can read audit logs.
	notifications — Own Notifications
	POLICY: SELECT, UPDATE for authenticated. USING: patron_id = auth.uid(). Patrons can only read and mark-read their own notifications.
	fines — Own Fines Read
	POLICY: SELECT for authenticated. USING: patron_id = auth.uid() OR auth.jwt()->>role IN (librarian, admin, superadmin).
	

10.3 Additional Security Requirements
* API rate limiting: implement via Supabase table (rate_limits) + Edge Function middleware. Limits: 100 searches/hour per IP, 20 holds/day per patron, 50 AI requests/day per patron.
* Input sanitization: all user text inputs sanitized with DOMPurify before storage.
* SQL injection: impossible via Supabase parameterized queries. Never use raw string interpolation in queries.
* CORS: Next.js API Routes only accept requests from own domain. Supabase CORS configured to project URL only.
* Secrets management: all API keys in Vercel environment variables (not committed to git). Supabase Vault used for NIM API key accessed only from Edge Functions.
* Content Security Policy: strict CSP headers via next.config.js. No inline scripts.
* File upload security: Supabase Storage policies restrict uploads to authenticated staff only, max 10 MB, allowed MIME types: image/jpeg, image/png, image/webp, application/epub+zip, audio/mpeg.

11. Performance & Scalability Strategy
11.1 Caching Architecture
Data Type
	Caching Strategy
	Book catalog pages
	ISR (Incremental Static Regeneration) with 60-second revalidation. revalidatePath() called on any catalog update. CDN-cached on Vercel Edge.
	Book detail pages
	ISR with 300-second revalidation. generateStaticParams() pre-builds top 1,000 books by popularity.
	AI summaries
	Cached in Supabase ai_summary column. TTL: 90 days. Cache key: book_id + model version.
	AI recommendations
	Cached in patron ai_reading_profile JSONB. Recomputed daily via Supabase Cron at 3 AM UTC.
	Search results
	Client-side cache via SWR (stale-while-revalidate). TTL: 30 seconds. Deduplicated concurrent requests.
	Book availability
	No caching — always real-time via Supabase Realtime subscription on book.available_copies.
	External API responses
	Open Library, Google Books, Wikipedia: cached in Supabase (external_api_cache table) for 30 days. Saves API calls and improves speed.
	Images
	Next.js Image Optimization with Vercel CDN. Unsplash images proxied through Next.js to enable caching. All images converted to WebP.
	

11.2 Database Query Optimization
* Indexes: Create indexes on all FK columns, status columns, and date columns used in queries. Full-text search index on books(title, authors, description) using pg_trgm.
* Pagination: All list queries use cursor-based pagination (not OFFSET). Page size: 20 items default, 50 maximum.
* N+1 Prevention: All queries that fetch books with related data use JOIN or Supabase select() with nested selects. Never fetch books then individually query for each.
* Connection pooling: Supabase uses PgBouncer in transaction mode. Use Supabase client (not direct Postgres URL) in all Next.js Route Handlers to leverage pooling.
* pgvector: IVFFlat index on book embeddings with lists=100 for efficient ANN search.
